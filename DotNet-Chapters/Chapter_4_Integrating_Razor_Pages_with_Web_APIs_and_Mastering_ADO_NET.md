# Chapter 4: Integrating Razor Pages with Web APIs and Mastering ADO.NET

## Part I: Demonstrate Razor UI and API Integration

# 

In modern web architecture, it is common to separate the user interface (UI) from the data and business logic. A powerful pattern in ASP.NET Core is to build a server-rendered UI using Razor Pages that consumes a separate, data-centric Web API. This creates a decoupled system where the UI (the Razor app) acts as a client to the API, which can also serve data to other clients like mobile apps or single-page applications (SPAs).

This section demonstrates how to integrate a Razor Pages application with a backend Web API for all its data needs.

### Section 1: Configuring the Razor Pages Client

# 

To communicate with a Web API, the Razor Pages application needs an HTTP client. The recommended approach in ASP.NET Core is to use IHttpClientFactory to manage HttpClient instances efficiently.

1.     Configure appsettings.json:  
First, store the base URL of the Web API in the appsettings.json file. This avoids hardcoding URLs and makes it easy to change the API endpoint for different environments (development, production, etc.).  
JSON  
{  "Logging": {... },  "AllowedHosts": "\*",  "ContactsApi": "https://localhost:5001"  
}  
  

2.     Register HttpClient in Program.cs:  
Next, register a typed or named HttpClient in the dependency injection (DI) container. This is done in the Program.cs file. Using AddHttpClient configures the client with the base address from our settings file and manages its lifetime effectively.  
C#  
```csharp
// In Program.cs  
var builder = WebApplication.CreateBuilder(args);// Add services to the container.  
builder.Services.AddRazorPages();  
  
// Register a typed HttpClient for the Contacts APIbuilder.Services.AddHttpClient<IContactsClient, ContactsClient>(client =>  
{    var apiBaseUrl = builder.Configuration.GetSection("ContactsApi").Value;  
    client.BaseAddress = new Uri(apiBaseUrl);  
});var app = builder.Build();//... middleware configuration...  
app.Run();  
 ``` 
_(Note: The IContactsClient and ContactsClient in this example would be a typed client generated by a tool like NSwag, which provides a strongly-typed way to call the API. A simpler approach could use a named client.)_

### Section 2: Consuming the API in the PageModel

# 

With the HttpClient configured, you can now inject it into your PageModel classes to fetch and send data. The PageModel acts as the intermediary, calling the API and preparing the data for the Razor view.

The following example shows an Index.cshtml.cs PageModel that fetches a list of contacts from the API.

C#
```csharp
// In Pages/Contacts/Index.cshtml.cs  
public class IndexModel : PageModel{    
    // Using a typed client for API access  
      private readonly IContactsClient client;    
    // // The client is injected via the constructor by the DI container    
    public IndexModel(IContactsClient client)    
    {  
        client = client;  
    }    
// A public property to hold the list of contacts for the view    
    public IList<Contact> Contact { get; set; }    
    // The OnGetAsync handler is called when the page is requested    
    public async Task OnGetAsync()    {        
        // Call the API to get the list of contacts and assign it to the property        
        Contact = (await client.GetContactsAsync()).ToList();  
    }  
}
```


In this PageModel:

●       The IContactsClient (our typed HttpClient) is injected into the constructor.

●       The OnGetAsync handler method is executed when a GET request is made to the page.

●       Inside OnGetAsync, we call the API's GetContactsAsync method. The returned data is deserialized from JSON into a list of Contact objects.

●       This list is assigned to the public Contact property, making it available to the corresponding .cshtml view file.

### Section 3: Displaying API Data in the Razor View

# 

The final step is to render the data in the .cshtml view. The view is strongly typed to its PageModel (using the @model directive), giving it access to all public properties, including the list of contacts we fetched from the API.

HTML
```html
@page  
@model RazorApiIntegration.Pages.Contacts.IndexModel  
 
@{  
    ViewData = "Contacts";  
}  
  
<h1>Contacts</h1>  
  
<p>    
<a asp-page="Create">Create New</a>  
</p>  
<table class="table">    
<thead>     
   <tr>           
        <th>@Html.DisplayNameFor(model => model.Contact.Name)</th>            
        <th>@Html.DisplayNameFor(model => model.Contact.Email)</th>            
        <th></th>        
    </tr>    
    </thead>    
    <tbody>@foreach (var item in Model.Contact){        <tr>            
    <td>@Html.DisplayFor(modelItem => item.Name)
    </td>            
    <td>@Html.DisplayFor(modelItem => item.Email) 
    </td>            
    <td>
    <a asp-page="./Edit" asp-route-id="@item.Id"\>Edit</a>|<a asp-page="./Details" asp-route-id="@item.Id"\>Details</a>|<a asp-page="./Delete" asp-route-id="@item.Id"\>Delete</a></td>        
    </tr>}</tbody>  
</table>  
  ```

This pattern can be extended to all CRUD (Create, Read, Update, Delete) operations. For example, a "Create" page would have an OnPostAsync handler that takes the form data (via model binding), packages it, and sends it to the API's POST endpoint using a method like \_client.PostContactAsync(contact). This architecture effectively separates the concerns of UI rendering (Razor Pages) from data management (Web API).

## Part II: Recap ADO.NET with PostgreSQL

# 

While high-level Object-Relational Mappers (ORMs) like Entity Framework Core are popular, it is essential to understand the foundational data access technology they are built upon: **ADO.NET**. ADO.NET provides a set of classes for interacting directly with a data source. It offers fine-grained control over database operations and can be more performant in certain scenarios.

This section provides a recap of ADO.NET, focusing on its use with PostgreSQL via the **Npgsql** data provider.

### Section 1: What is ADO.NET and Npgsql?

# 

**ADO.NET** is the core data access framework within.NET. It consists of a set of classes that expose data access services for.NET applications. It is "provider-based," meaning you use a specific data provider library to connect to a particular database.

**Npgsql** is the official, open-source ADO.NET data provider for PostgreSQL. It allows any.NET application to connect and interact with a PostgreSQL database server. To use it, you must first install the

Npgsql NuGet package into your project.

### Section 2: Core ADO.NET Objects for PostgreSQL

# 

The Npgsql provider implements the standard ADO.NET interfaces with PostgreSQL-specific classes. The five core objects you will work with are:

1.     **NpgsqlConnection**: Represents a unique session to a PostgreSQL database. Its primary job is to manage the physical network connection. You create it with a connection string and must explicitly open and close it (preferably with a using statement to ensure it's properly disposed).

2.     **NpgsqlCommand**: Represents a SQL statement (SELECT, INSERT, UPDATE, DELETE) or a stored procedure to be executed against the database. Key properties include Connection, CommandText (the SQL query), and Parameters.

3.     **NpgsqlDataReader**: Provides a high-performance, forward-only, read-only stream of data from a SELECT query. It is highly efficient for reading query results as it does not load the entire result set into memory at once.

4.     **NpgsqlParameter**: Represents a parameter to an NpgsqlCommand. Using parameters is **critical** for preventing SQL injection attacks. Instead of concatenating user input into a query string, you use placeholders (e.g., @name, @id) and add NpgsqlParameter objects to the command's Parameters collection.

5.     **NpgsqlTransaction**: Represents a database transaction, allowing you to group multiple database operations into a single atomic unit of work.

### Section 3: A Practical CRUD Workflow with Npgsql

# 

The following examples demonstrate how to perform basic Create, Read, Update, and Delete (CRUD) operations using Npgsql and ADO.NET.

#### Connecting to the Database

# 

All operations begin by establishing a connection. The connection string contains the server address, database name, and credentials.

C#
```csharp
string connectionString = "Host=localhost;Username=myuser;Password=mypassword;Database=mydatabase;";  
await using var connection = new NpgsqlConnection(connectionString);await connection.OpenAsync();
```
#### Create (INSERT)

# 

To insert a new record, use ExecuteNonQueryAsync(), which returns the number of rows affected.

C#
```csharp
await using var cmd = new NpgsqlCommand("INSERT INTO employee (first\_name, last\_name) VALUES (@fname, @lname)", connection);  
  
cmd.Parameters.AddWithValue("@fname", "John");  
cmd.Parameters.AddWithValue("@lname", "Doe");  
  
int rowsAffected = await cmd.ExecuteNonQueryAsync();  
Console.WriteLine($"{rowsAffected} row(s) inserted.");  
  
````
#### Read (SELECT)

# 

To read data, use ExecuteReaderAsync() and loop through the results with the NpgsqlDataReader.

C#
```csharp
await using var cmd = new NpgsqlCommand("SELECT first\_name, last\_name FROM employee", connection);  
await using var reader = await cmd.ExecuteReaderAsync();while (await reader.ReadAsync())  
{  
    Console.WriteLine($"Name: {reader.GetString(0)} {reader.GetString(1)}");  
}  
  
```
#### Update (UPDATE)

# 

Updating data is similar to inserting; you use a parameterized query and ExecuteNonQueryAsync().

C#
```csharp
await using var cmd = new NpgsqlCommand("UPDATE employee SET first\_name = @fname WHERE id = @id", connection);  
  
cmd.Parameters.AddWithValue("@id", 1);  
cmd.Parameters.AddWithValue("@fname", "Jane");  
  
int rowsAffected = await cmd.ExecuteNonQueryAsync();  
Console.WriteLine($"{rowsAffected} row(s) updated.");  
```  

#### Delete (DELETE)

# 

Deleting follows the same pattern, using a DELETE statement with a WHERE clause.

C#
```csharp
await using var cmd = new NpgsqlCommand("DELETE FROM employee WHERE id = @id", connection);  
  
cmd.Parameters.AddWithValue("@id", 1);  
  
int rowsAffected = await cmd.ExecuteNonQueryAsync();  
Console.WriteLine($"{rowsAffected} row(s) deleted.");  
```  

By mastering these core ADO.NET concepts with Npgsql, you gain the ability to perform any database operation with PostgreSQL, providing a solid foundation for data access in any.NET application.
